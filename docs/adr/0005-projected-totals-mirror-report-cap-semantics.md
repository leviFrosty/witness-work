# Projected Totals mirror report cap semantics; Plans are typed by Category reference

The original Projected Total design applied the credit cap as a blanket clamp on all planned minutes (`planned = min(planned, cap − logged)`), which silently ceilinged every projection at the cap — a regular pioneer planning a 70-hour month of ordinary ministry saw 55h. The clamp was deliberate (Plans were untyped, so the conservative reading treated them all as potentially-credit), but it punished the overwhelmingly common case: planned time is ordinary house-to-house preaching, which the cap never touches.

We decided a **Projected Total is the adjusted total the Service Report would show if every remaining Plan became reality** — computed by combining logged and planned time into Standard and Credit buckets and running the _same_ cap formula a finished month gets (`combinedStandard > cap ? combinedStandard : min(combinedStandard + combinedCredit, cap)`), month by month, since the month is the unit the cap governs. We rejected the naive alternative (`loggedAdjusted + planned`, capping only planned credit) because it can display a number the report will never reach: planned standard time that pushes combined standard past the cap retroactively squeezes out _logged_ credit, and only the mirror formula accounts for that. The displayed "planned" portion of a projection is therefore the Plans' effective contribution (`projected − logged`), not their raw sum; standalone planned-minutes displays elsewhere remain raw intent and are never cap-adjusted.

To let the projection bucket planned time honestly, **Day Plans and Recurring Plans carry an optional `categoryId`** — the same Category reference Time Entries use — and nothing else: no stamped `credit` boolean (unlike Time Entries, which keep one for legacy readers). The Category record is the single source of truth for a Plan's credit-ness, derived at read time, so flipping a Category's credit setting retroactively re-forecasts every Plan referencing it with no re-stamping pass. A Plan with no `categoryId`, or whose Category has been deleted, forecasts Standard — which also makes every pre-existing Plan Standard by construction, with zero migration. Type is pattern-level on Recurring Plans (Overrides cannot change it; skip the instance and create a Day Plan instead), and Assistant Recommendations only ever emit Standard plans, because a credit recommendation near the cap would propose hours that contribute nothing toward the goal gap the Assistant exists to close.
