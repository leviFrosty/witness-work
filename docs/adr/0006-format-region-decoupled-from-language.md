# Format Region is decoupled from Language; date conventions are patched onto the language locale, not selected as a locale

Historically a single `locale` preference did double duty: it picked the translation Language _and_ set moment's global locale, so date ordering, 12/24-hour time, and first-day-of-week were all a side effect of which language the User read the app in. That coupling has no escape hatch — an Australian who wants the app in English is forced into `en-us` → moment `en` → `MM/DD/YYYY`, with no way to get `DD/MM/YYYY` short of reading the app in a different language. The recurring request ("let me change the date format") is really a request to **separate formatting conventions from translation language**.

We decided to model a **Format Region** as a first-class preference independent of **Language**. Language continues to drive the words the User reads — including month and weekday _names_; Format Region drives _ordering_ (date order), _clock_ (12 vs 24-hour), and the default _first day of week_. The obvious implementation — `moment.locale(region)` — is **wrong here and is explicitly rejected**, because moment's locale is a single global that owns both the names and the ordering: switching it to `en-au` to get Australian date order would also drag the month names back to English for a Spanish reader. moment cannot serve Spanish names and Australian order from one global locale. Instead we keep `moment.locale` pinned to the **Language** and overlay the Region's conventions onto it with `moment.updateLocale(languageLocale, { longDateFormat, week: { dow } })`, reading the Region's `longDateFormat` and `firstDayOfWeek` from moment's own already-bundled locale data first (no new dependency, no curated CLDR table). Names then come from the language locale's `months`/`weekdays` arrays while order comes from the patched `longDateFormat` tokens — and the `LT` token (`h:mm A` vs `HH:mm`) carries 12/24-hour for free. This forces a **read-time convention**: every user-facing date must render through the locale-aware tokens (`L`/`ll`/`LL`/`LT`/`LLL`) rather than hardcoded `MMM D, YYYY`-style strings, or the patch never reaches it; ISO storage keys (`YYYY-MM-DD`) stay hardcoded because they are not display. Tight headings that deliberately omit the year (`MMM D`, `ddd, MMM D`) route through a sanctioned compact helper that flips day/month order by Region while staying short, mirroring the `formatMinutesCompact` bypass.

The Region bundle sets defaults that three single-setting rows — Start of Week, Time Format, Date Order — may each override, with precedence **row override → explicit Region's moment locale → device**. At the device rung, resolution is hybrid per axis: Time Format and Start of Week come from `getCalendars()[0]` (`uses24hourClock`, `firstWeekday`) so a manual iOS 24-hour toggle is honored, while Date Order comes from the device locale's moment `L` pattern since no calendar API exposes ordering. The Date Order override governs only the numeric forms (`L`/`l`) and the compact helper's order; written-out month-name dates keep Region ordering, because `YMD` with a month name (`2026 Jun 10`) reads as broken. Existing installs all carry `startOfWeek: 0` from the old hardcoded default; the persist migration converts a stored `0` to Auto (so Region finally drives it — US users see no change, UK/EU users get Monday automatically) while preserving any `1–6` as a deliberate explicit override. The widget snapshot inherits the patch because it is built in the app's JS runtime, with the one caveat that the background-refresh task must re-apply Language + Region to moment before rebuilding or its dates revert to US format.
