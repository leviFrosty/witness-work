# Profile overlay uses a persistent-mount pattern via FullWindowOverlay

The expanding profile overlay (`ProfileDetailOverlay`) contains two heavy subtrees — `ContributionGraph` (~180 cells with a gesture detector) and `MonthlyRoutine` (a horizontal FlashList) — whose mount cost previously caused either spring stutter (when rendered during the open animation) or a visible "content snap" (when deferred behind a `contentReady` gate that flipped true after the spring landed). To eliminate the snap entirely, the overlay now mounts persistently on home idle: `ProfileCard` triggers `InteractionManager.runAfterInteractions` to measure its anchor and seed the overlay's `origin` prop, after which the overlay's host renders unconditionally and its heavy children render once at expanded dimensions. The open spring is then a pure opacity + container morph against an already-rendered tree, with no React reconciliation or layout work to compete with the animation. The host is `<FullWindowOverlay>` from `react-native-screens` rather than `<Modal>` because RN's `<Modal>` absorbs touches at the UIWindow level regardless of inner `pointerEvents`, which we confirmed empirically by breaking the entire app for one iteration — `FullWindowOverlay` renders into its own UIWindow whose hit-test correctly passes transparent areas through to the underlying app window, which is required for a closed-but-mounted overlay not to eat home-screen taps. We rejected three alternatives: smoothing the snap with a Reanimated `FadeIn` entering animation (good but not zero pop), pre-rendering the heavy children in a hidden offscreen `View` on `HomeScreen` (fresh instances still mount inside the overlay), and adding `react-native-portalize` for true instance sharing (already-installed `react-native-screens` ships the exact primitive). `FullWindowOverlay` is iOS-only, which matches the app's iOS-only target.
